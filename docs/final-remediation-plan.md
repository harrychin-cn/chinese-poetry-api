# 诗词画曲赋 — 最终整改方案

> 版本：v1.0 | 2026-07-04  
> 基于完整代码审计 + 控制台 UX 审查 + 用户目标确认

---

## 一、目标重述

**做一个可收费的 SaaS 产品，跑出第一个付费客户。** 让喜爱诗词曲赋的人找到归属，同时获得应得的收入。

核心约束：零营销预算，以产品体验和内容质量驱动增长。

---

## 二、当前状态诊断

### 2.1 技术债总结

| 问题 | 严重度 | 影响 |
|------|--------|------|
| 控制台左侧布局混乱——7 个功能卡片堆叠，无层级 | P0 | 用户打开后困惑，转化率极低 |
| 作品创作表单占据侧栏最大面积 | P0 | 次要功能抢占核心操作区 |
| 移动端无适配 | P0 | 微信朋友圈分享后无法正常使用 |
| 功能范围失控——原创作品/音频/音乐/证书/区块链/钱包/打赏全部实现 | P1 | 零客户阶段建了 11 个模块，资源分散 |
| 黄金评测集由 Codex 代理标注，非真实人工 | P1 | AI 审查 AI，质量基准不可靠 |
| 零商业验证即宣布开发完工 | P1 | 核心计费链路未在生产环境跑通 |
| API Key 可从 URL 查询参数读取 | P2 | 安全风险 |
| Go 1.25.0 预发布版 | P2 | 构建不稳定 |

### 2.2 控制台 UX 问题清单

当前 [console.html](file:///G:/项目/唐诗宋词数据库API/chinese-poetry-api/internal/api/rest/handler/console.html)（1497行）是一张单页面，左侧侧栏垂直堆叠 7 个卡片：

1. **品牌 Logo** — 纯装饰，占黄金位
2. **API Key 与额度** — 合理
3. **账号与原创作品库** — 最大问题：完整 CRUD 表单 + 反向创作 + 9 个按钮，占整个侧栏 40% 以上面积
4. **自然语言搜索诗词** — 核心功能被埋在第三个卡片，需滚动
5. **Qanlo 生图 Key** — 合理但位置靠后
6. **Qanlo 绑定/充值** — 合理
7. **Wallet/积分/打赏** — 零客户阶段完全不需要

**右侧主区利用率极低**：占 68% 屏幕宽度，在用户未搜索时只有一句引导语和背景图。

**移动端完全不可用**：`@media` 媒体查询只做了简单响应，无触控优化，无 PWA。

## 三、整改路线图

### 阶段 A：控制台重设计（P0，本周完成）

目标：让用户 10 秒内理解"我在这里能做什么"，30 秒内完成第一次搜索。

**不改后端**，纯前端重写 [console.html](file:///G:/项目/唐诗宋词数据库API/chinese-poetry-api/internal/api/rest/handler/console.html)。

#### A1. 布局改为经典中式 SaaS 三栏

```
┌───────────────────────────────────────────────────────┐
│  顶栏：诗词画曲赋 | 首页 | 控制台 | 作品库 | 文档 | 价格  │
├──────────┬────────────────────────────────┬───────────┤
│ 左侧菜单  │        主工作区               │  状态栏    │
│          │                               │           │
│ 诗词搜索  │  ┌─────────────────────────┐  │ API Key   │
│ 我的作品  │  │ 搜索框 + 场景标签        │  │ 状态指示  │
│ 用量统计  │  │                         │  │ 今日用量  │
│          │  │ 诗词卡片  │  意境图区     │  │ 快捷充值  │
│          │  │          │              │  │           │
│          │  └─────────────────────────┘  │           │
│          │  相关推荐                      │           │
│          │  图片 Prompt 区               │           │
├──────────┴────────────────────────────────┴───────────┤
│  底部：Powered by Qanlo · 诗词画曲赋 © 2026            │
└───────────────────────────────────────────────────────┘
```

**核心改动**：

1. **顶栏导航**：品牌名 + 5 个入口（首页/控制台/作品库/文档/价格），固定不滚动
2. **左侧菜单 3 项**：诗词搜索、我的作品、用量统计。不是表单容器，是功能导航
3. **主工作区**：默认就是搜索页，打开即可用。搜索框 + 场景标签 + 诗词结果左半 + 意境图右半
4. **右侧状态栏**：紧凑显示 API Key 状态、今日用量/限额、快捷充值入口、生图 Key 管理
5. **我的作品**：点击左侧菜单后，主工作区切换到完整作品管理（替代当前侧栏里那个庞大表单）
6. **底部栏**：版权 + Powered by Qanlo

#### A2. 默认主工作区 = 搜索 + 结果显示

打开 https://qanlo.com/poetry-api/ 直接看到搜索框和默认示例诗词，不需要先配置任何东西。

#### A3. 裁掉"零客户不需要"的功能入口

| 裁掉内容 | 理由 |
|----------|------|
| Wallet/积分/打赏卡片 | 零客户，零交易 |
| 配乐草稿生成按钮 | 零需求 |
| Issue Certificate / Open Certificate | 零客户 |
| 反向创作入口 | 降到"我的作品"页的可选功能 |
| 作品资产展示区 | 降到"我的作品"页 |

缩减后，控制台只保留核心路径：搜索诗词 → 查看结果 → （可选）生成意境图 → （可选）保存为自己的作品。

#### A4. PWA + 移动端优先

- 添加 `manifest.json`，支持添加到手机主屏幕
- 添加 Service Worker 做离线缓存
- `viewport` 已设置，补 `apple-mobile-web-app-capable` 等 meta 标签
- 所有按钮最小触控区域 ≥ 44×44px
- 诗词卡片在移动端单列显示，意境图区可折叠

### 阶段 B：功能精简（P1，两周内）

目标：砍掉过早投资的功能模块，聚焦核心价值。

#### B1. 后端路由标记为"已实现但前台不暴露"

不改代码，只在 HTML 层面隐藏以下入口（API 路由仍保留，防止已有外部调用）：

| 功能 | 处理方式 |
|------|----------|
| `/api/v1/works/:id/audio/generate` | 隐藏入口 |
| `/api/v1/works/:id/music/generate` | 隐藏入口 |
| `/api/v1/works/:id/certificate` | 隐藏入口 |
| `/api/v1/works/:id/certificate/anchor` | 隐藏入口 |
| `/api/v1/wallet/*` | 隐藏入口 |
| `/api/v1/works/:id/tip` | 隐藏入口 |
| `/api/v1/works/reverse-create` | 降到作品页次要入口 |
| `/api/v1/partners/works/export` | 隐藏入口 |

#### B2. 首页改版

当前 `/console` 是唯一的用户入口。新增一个 `/` 或 `/home` 首页，作为产品门面：

- 古风视觉 + 一句话定位："AI 驱动的诗词知识库"
- 三个核心功能卡片：搜诗词 / 赏意境 / 创作品
- 一键跳转控制台（无需注册即可试用基础搜索）

---

### 阶段 C：质量底线（P1，一个月内）

#### C1. 黄金评测集人工标注

当前 1000 条黄金评测集由 Codex 代理标注。需要你亲自抽检至少 50 条：

- 从 `data/enrichment/golden-sample-1000.reviewed.jsonl` 随机取 50 条
- 逐条检查：标签是否合理、证据句是否准确、是否过度解读
- 统计"AI 标注 vs 人工判断"的一致率
- 如果一致率 < 85%，整个黄金集需要回炉

#### C2. 安全修复

1. 禁掉 API Key 从查询参数读取（`middleware/apikey.go` 去掉 `c.Query("api_key")` 分支）
2. Admin 端点加 IP 白名单或至少限流
3. 生产环境关闭 GraphQL introspection
4. Go 版本降到 1.23.x 稳定版

---

### 阶段 D：商业验证（P1，两个月内）

#### D1. 找 3 个微信好友试用

从你的几十个企业客户朋友圈里，私聊邀请 3 个人试用：

- 给他们发放一个每日限额 50 次的 Key
- 让他们只用手机打开——这是最终用户的主要场景
- 观察他们：能不能找到想搜的诗？会不会用生图？有什么困惑？
- 记录至少 5 条具体的体验问题

#### D2. 根据反馈迭代一轮

用真实反馈重新排优先级。不要猜测用户要什么，让用户告诉你。

#### D3. 朋友圈分享功能

加一个"分享这张诗词图"按钮，生成一张带二维码的分享卡片（含诗词 + 意境图 + 小程序风格的卡片），方便用户在朋友圈传播。这是零预算下唯一的增长引擎。

---

### 阶段 E：收费上线（P2，第三个月起）

前提：阶段 D 完成后，至少有 1 个人表达了"愿意付费"的意向。

1. 配置真实的 `QANLO_AGENT_APP_ID`
2. 开通精简充值链路
3. 免费层：每日 20 次搜索 + 3 次生图
4. 付费层：月卡 ¥29，每日 200 次搜索 + 30 次生图
5. 靠朋友圈裂变 + 口碑传播自然增长

## 四、立即执行清单（本周）

按优先级排序，每一项都可以独立完成：

5. **Go 版本降级** — 1.23.x

---

## 五、停止线

以下情况必须停下来：

1. 控制台改版后仍无真实用户试用 → 停止功能开发，先解决获客
2. 3 个月内没有任何人表达付费意向 → 重新评估产品方向
3. 数据增强标签一致率 < 85% → 停止发布高置信标签，回炉黄金集

## 六、与原规划文档的关系

本方案**覆盖** `development-plan.md` v1.6 和 `ultimate-product-roadmap.md` 的执行优先级。

原规划文档的架构设计、API 规格、数据模型仍作为技术参考保留。但**执行顺序重新排序为**：

1. 用户体验（控制台 + 移动端）— 当前缺失
2. 质量底线（人工抽检黄金集）— 当前 AI 自循环
3. 商业验证（真实用户试用）— 当前后置
4. 付费转化 — 当前未启动

原规划中已完成但过早投资的功能模块（音频/音乐/证书/区块链/打赏/合作方导出）**暂不删除代码，但不暴露入口**，等有付费客户后再评估是否开放。
