# 产品包装与收费方案

## 产品定位

增强版古诗词 API：给教育 App、小程序、内容工具、AI 知识库提供可商用的诗词检索服务。

它不是简单复制公开数据，而是在公开数据之上提供：

- 复合查询：作者、朝代、体裁、关键词、句数、字数组合筛选。
- 全文搜索：基于 SQLite FTS5 的相关性排序与命中摘要。
- 增值标签：主题、年级、场景、情绪等可人工校正标签。
- 商业底座：API Key、每日额度、用量统计、管理员接口。
- 私有部署：Docker 服务，可放到客户自己的服务器。

## 首版价格档（匹配 QanloAPI 按调用收费）

当前不自建收银台、订单和订阅系统。首版商业化只做两件事：

1. 本项目负责 API Key、套餐标记、每日限额、用量统计和接口保护。
2. 充值与扣费沿用 QanloAPI 精简链路：客户拿到管理员或 Qanlo 开通链路发放的 Key 后跳转充值页，后续按可计费 API 调用消耗余额。

> 下面价格是首版验证价，用来跑通“有人愿意为诗词检索 API 付费”这件事；不是长期正式价。

| 档位 | `tier` 标记 | 适合客户 | 建议额度/限制 | 收费方式 |
|---|---|---|---|---|
| 人工体验 | `trial` | 个人开发者、演示客户 | 每日 100 次增强/知识库调用，7-14 天试用 | 需管理员或 Qanlo 开通链路发放 Key，不公开自助免费生成 |
| 开发者包 | `developer` | 小程序、课程工具、内容站 | 建议每日 1,000-10,000 次；先给较低限额，稳定后人工上调 | QanloAPI 充值后按调用消耗，建议首充 99 元起 |

### 套餐落地规则

- `tier` 只是套餐和运营标记，不代表本项目已经内置支付订阅。
- `daily_limit` 控制每日最大调用量；余额和充值状态通过 `/api/v1/billing/status` 展示给客户。
- 客户余额不足或需要续费时，调用 `/api/v1/billing/qanlo/recharge-session` 获取 Qanlo 精简充值页。
- Qanlo 回调 `/api/v1/billing/qanlo/callback` 完成后，本项目刷新绑定状态和可用状态。
- 管理员可用 `PATCH /api/v1/admin/api-keys/:id` 人工调整 `tier`、`daily_limit`、启停状态和备注。

## 典型卖点

### 教育 App

- 按年级、主题、体裁筛选诗词。
- 支持“李白 + 月亮 + 小学”这类复合查询。
- 可以接入背诵、赏析、题库、作文素材模块。

### 小程序 / 内容工具

- 按节日、节气、文旅、短视频文案场景取诗句。
- 用 API Key 控制客户额度，方便按量售卖。

### AI 知识库 / RAG

- 先用标签和结构化字段缩小范围。
- 再用全文搜索做召回。
- 返回作者、朝代、体裁等结构化信息，方便模型引用。

## 10 分钟接入路径

1. 部署服务：`docker compose up -d`。
2. 设置管理员密钥：`API_ADMIN_TOKEN=你的密钥`。
3. 管理员创建/发放客户 API Key，并写入 `tier`、`daily_limit`。
4. 客户调用 `/api/v1/billing/qanlo/provision` 绑定 Qanlo Key。
5. 余额不足或主动续费时，调用 `/api/v1/billing/qanlo/recharge-session` 跳转 Qanlo 充值页。
6. 客户调用 `/api/v1/poems/query`、`/api/v1/poems/search/fulltext` 或 `/api/v1/knowledge/recall`。
7. 用 `/api/v1/billing/status` 和 `/api/v1/admin/api-keys` 查看余额/绑定状态和今日用量。

## 当前交付边界

已经具备可演示、可私有化试点的商业闭环：

- 有增强查询能力。
- 有全文搜索能力。
- 有标签增值入口。
- 有 API Key 和额度控制。
- 有 Docker 部署基础。

## 商业验证节奏

首版目标不是先做完整 SaaS 支付后台，而是用现有 QanloAPI 链路验证真实付费意愿：

1. 找 3-5 个教育 App、小程序或内容工具试用；如需试用，由管理员发放 `trial` Key，不公开自助免费领用。
2. 观察 7 天内的调用量、热门查询、错误率和客户反馈。
3. 对有持续调用的客户，引导 Qanlo 首充 99 元或 999 元，验证是否愿意付费。
4. 付费后只人工调整 `tier` 和 `daily_limit`，不新增自建订单系统。
5. 若至少 2 个客户持续付费，再考虑补正式价格页、合同/发票和私有部署报价。

后续真正卖给客户前，建议继续补：

- 更大批量的人工校正标签数据。
- 更完整的文档站页面、在线试用页和客户案例。
- 正式合同/发票/私有部署报价材料。
- 慢查询日志和生产监控面板。
