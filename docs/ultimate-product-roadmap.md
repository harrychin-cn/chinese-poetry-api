# 诗词画曲赋：终极产品形态与长期迭代路线图

> 目标：把当前“AI 诗词知识库 API”升级为一个长期沉淀、可创作、可检索、可生成图像/音乐、可交易、可确权识别的诗词曲赋平台。

## 1. 终极愿景

本项目最终不是单纯的诗词查询 API，而是：

> **世界级诗词曲赋创作与沉淀平台。**

用户可以在平台里：

1. 创作并录入自己的诗、词、曲、赋。
2. 为作品生成专属意境图。
3. 为作品生成吟诵、配乐、甚至完整诗词音乐。
4. 通过图片、故事、意境描述反向生成诗词曲赋。
5. 将自己的作品纳入公共诗词曲赋库。
6. 通过积分、余额、打赏、交易等方式流转作品价值。
7. 为原创作品生成唯一识别码，后期可接区块链存证。

核心资产从“古代诗词数据”扩展为：

- 古代诗词曲赋库；
- 现代用户原创作品库；
- 作品意境图；
- 作品吟诵音频；
- 作品配乐与歌曲；
- 作品相似度与原创性证据；
- 作品版本、作者、授权、交易与存证记录。

## 2. 产品底层原则

### 2.1 平台沉淀优先

用户创作不是一次性生成，而是进入可持续沉淀的作品库。

每个作品都要有：

- 作品正文；
- 作者/提交者；
- 创作时间；
- 作品类型：诗、词、曲、赋、现代诗、歌词等；
- 授权状态；
- 原创查重记录；
- 版本记录；
- 图片/音乐等衍生资产；
- 唯一作品编号。

### 2.2 版权放弃不等于允许抄袭

平台可以设计“主动开放版权 / 公共使用授权”机制，但必须同时要求：

1. 用户承诺提交内容是自己原创，或有合法授权。
2. 用户同意将作品以平台规定的开放方式发布。
3. 平台保留查重、下架、冻结交易、标记争议的权利。
4. 抄袭作品即使被提交，也不能因为点击协议而变成合法作品。

产品协议建议采用“CC0-like / 公共领域贡献”思路：尽可能让用户放弃作品经济权利，允许他人复制、改编、商用、传播。但不同法域对作者署名权、保护作品完整权等精神权利处理不同，所以正式上线前协议文本必须单独做法务审查。

### 2.3 先查重，再发布，再交易

作品进入公共库和交易系统前，必须经过原创性检查：

```text
用户提交作品
  ↓
文本规范化
  ↓
精确重复检测
  ↓
近似文本查重
  ↓
语义相似检测
  ↓
风险分级
  ↓
通过 / 需修改 / 人工复核 / 拒绝发布
```

### 2.4 AI 是创作助手，不是数据黑洞

AI 生成的诗词、图片、音乐都必须记录生成来源：

- 用户原始输入；
- 使用模型；
- prompt；
- 生成时间；
- 用户是否二次修改；
- 最终发布版本。

后期查重、交易、争议处理都依赖这些记录。

## 4. 核心产品流程

### 4.1 用户录入原创作品

```text
用户注册/登录
  ↓
填写作品标题、类型、正文、说明
  ↓
勾选原创承诺 + 开放版权协议
  ↓
系统查重
  ↓
生成作品 ID
  ↓
可选择生成意境图/吟诵/配乐
  ↓
发布到公开库或保存为私密作品
```

### 4.2 根据诗词生成意境图

生图不能随意配图，必须绑定诗词意境：

```text
作品正文
  ↓
提取核心意象：月、江、山、云、花、柳、舟、酒、楼等
  ↓
判断氛围：清冷、思念、辽阔、明亮、悲凉、豪迈等
  ↓
生成图片 Prompt
  ↓
调用 gpt-image-2 或其他图片模型
  ↓
保存图片资产与生成参数
```

图片资产必须记录：

- 来源作品 ID；
- prompt；
- 模型；
- 图片尺寸；
- 生成时间；
- 是否公开；
- 是否可交易。

### 4.3 根据图片/故事生成诗词曲赋

```text
用户上传图片 / 输入故事 / 输入意境描述
  ↓
AI 解析主题、景物、情绪、人物、时代风格
  ↓
用户选择体裁：五绝、七绝、词牌、曲牌、赋、现代诗
  ↓
生成候选作品
  ↓
用户修改确认
  ↓
进入查重与授权流程
```

生成后必须提示用户：只有经过修改确认并通过查重，才能作为个人作品发布。

### 4.4 作品交易与积分

交易不应一开始做复杂市场，建议分三层：

1. **积分消耗**：生成图片、音乐、查重高级报告消耗积分。
2. **打赏**：用户给喜欢的作品或作者打赏。
3. **作品授权交易**：后期允许作品进入交易市场。

资金与积分流水要独立记录：

- 充值；
- 消耗；
- 打赏；
- 退款；
- 冻结；
- 争议处理；
- 平台分成。

## 5. 智能查重设计

### 5.1 查重对象

必须覆盖：

- 古代诗词曲赋库；
- 平台现代原创库；
- 用户草稿库中的公开候选；
- AI 生成历史；
- 常见网络语料快照；
- 已下架争议作品。

### 5.2 查重算法分层

| 层级 | 方法 | 作用 |
| --- | --- | --- |
| L1 | 文本规范化 + hash | 完全重复秒级拦截 |
| L2 | 分句、分词、n-gram overlap | 发现大段复制和改几个字的抄袭 |
| L3 | SimHash / MinHash | 快速近似查重 |
| L4 | embedding 语义相似 | 发现换词改写、语义高度相似 |
| L5 | 诗词格律/意象相似 | 针对诗词曲赋的专项判断 |
| L6 | 人工复核队列 | 高风险但不确定样本进入人工 |

### 5.3 风险分级

| 风险 | 处理 |
| --- | --- |
| 低风险 | 允许保存和发布 |
| 中风险 | 提示相似来源，要求用户修改或确认引用 |
| 高风险 | 禁止公开发布，进入人工复核 |
| 精确重复 | 直接拒绝发布 |
| 争议作品 | 冻结交易和公开展示 |

### 5.4 查重报告字段

每次查重要保存：

- `work_id`
- `normalized_hash`
- `simhash`
- `minhash`
- `embedding_model`
- `top_matches`
- `risk_level`
- `risk_reason`
- `review_status`
- `created_at`

## 6. 作品唯一识别码与区块链路线

不要一开始就上链，先做平台内唯一识别码。

### 6.1 第一阶段：平台作品 ID

格式示例：

```text
PCQF-2026-00000001
```

每个作品生成：

- 作品 ID；
- 版本 ID；
- 文本 hash；
- 作者 ID；
- 授权协议版本；
- 查重报告 ID。

### 6.2 第二阶段：证书页

每个公开作品有一个证书页：

```text
/works/PCQF-2026-00000001/certificate
```

显示：

- 作品标题；
- 作者；
- 发布时间；
- 授权方式；
- 文本指纹；
- 查重状态；
- 生成的图/音频资产；
- 交易和打赏状态。

### 6.3 第三阶段：区块链存证

上链内容只放摘要，不放完整正文：

- 作品 ID；
- 文本 hash；
- 授权协议 hash；
- 发布时间；
- 平台签名。

这样成本低，也避免把争议内容永久写入链上。

## 7. 迭代路线图

### 阶段 0：当前基础版，先稳住 API 和控制台

目标：让当前项目继续可用。

已具备或正在做：

- 诗词查询 API；
- 知识库召回；
- API Key；
- Qanlo 绑定 / 充值；
- 控制台；
- 意境图 Prompt；
- 数据增强路线从规则扩批改为 AI 候选 + 自动校验。

阶段结束标准：

- 本地 smoke 稳定通过；
- 控制台普通用户能看懂；
- 查询、充值、用量、反馈链路可跑通。

### 阶段 1：原创作品库 MVP

目标：允许用户录入自己的诗词曲赋。

功能：

- 用户账号；
- 作品创建、编辑、保存、发布；
- 作品类型：诗、词、曲、赋、现代诗；
- 原创承诺；
- 开放版权协议确认；
- 作品 ID；
- 公开/私密状态；
- 个人作品页。

后端表：

- `users`
- `works`
- `work_versions`
- `work_license_acceptances`
- `work_publication_events`

阶段结束标准：

- 用户可以提交一首原创作品；
- 系统能保存版本；
- 能公开展示；
- 能回看授权记录。

### 阶段 2：智能查重 MVP

目标：防止平台变成抄袭库。

功能：

- 文本规范化；
- 精确 hash 查重；
- n-gram 近似查重；
- 与古代诗词库比对；
- 与平台原创库比对；
- 风险报告；
- 高风险拦截。

后端表：

- `work_fingerprints`
- `similarity_matches`
- `plagiarism_reports`
- `manual_review_queue`

阶段结束标准：

- 完全复制古诗不能发布为原创；
- 大段改字能被提示；
- 高风险作品进入人工复核。

### 阶段 3：意境图生成

目标：每个作品都可以生成专属诗词图。

功能：

- 意象提取；
- 情绪提取；
- 图片 Prompt 生成；
- 图片模型配置：

```text
IMAGE_BASE_URL
IMAGE_MODEL=gpt-image-2
用户在生图页面填写自己的 Qanlo 生图 API Key
```

- 生成图片；
- 保存图片资产；
- 保存本地文件资产 URL 与 checksum；
- 重新生成；
- 下载图片。

后端表：

- `media_assets`
- `image_generation_jobs`
- `image_prompts`
- `credit_wallets`
- `credit_transactions`

阶段结束标准：

- 一首作品可以生成至少一张相关意境图；
- 图片必须引用原文意象；
- 生图消耗进入积分流水，缓存命中不重复扣分。

### 阶段 4：吟诵、配乐与诗词音乐

目标：让诗词回到“可吟唱”的形态。

功能：

- 文本转吟诵；
- 选择朗诵风格：男声、女声、古风、沉静、豪迈；
- 背景音乐；
- 按词牌/曲牌生成旋律草稿；
- 歌曲生成；
- 音频下载。

后端表：

- `audio_generation_jobs`
- `music_generation_jobs`
- `media_assets`（`asset_type=audio/music`，统一存储作品衍生资产）
- `credit_transactions`（吟诵真实生成扣积分；配乐草稿默认 0 分，可配置）

当前 MVP 进度：

- 作品级 `POST /api/v1/works/:id/audio/generate` 已支持 dry_run 和 OpenAI 兼容 `/audio/speech` 真实生成，成功后存本地音频资产；
- 作品级 `POST /api/v1/works/:id/music/generate` 先生成可复核 JSON 配乐草稿，后续再接正式音乐模型；
- 控制台作品管理区已能触发吟诵音频、配乐草稿，并在作品资产列表展示图片 / 音频 / 草稿。

阶段结束标准：

- 用户能为作品生成朗诵音频；
- 能选择配乐风格；
- 音频资产与作品绑定。

### 阶段 5：反向创作引擎

目标：用户可以从图片、故事、意境描述生成诗词曲赋。

功能：

- 图片理解；
- 故事解析；
- 意境描述解析；
- 体裁选择；
- 格律约束；
- 多候选生成；
- 用户二次修改；
- 查重后发布。

当前 MVP 进度：

- `POST /api/v1/works/reverse-create` 已支持从故事、意境、图片说明生成可编辑诗/词/曲/赋草稿；
- `save=true` 会把结果保存为私有原创作品草稿，不自动发布、不绕过原创承诺和授权确认；
- `GET /api/v1/works/reverse-jobs` 可查看当前 API Key 的反向创作任务记录；
- 控制台原创作品区已增加“反向创作”入口，可一键填入作品表单或保存为草稿。

阶段结束标准：

- 上传一张图能生成诗；
- 输入故事能生成词/赋；
- AI 作品不能绕过原创检查。

阶段结束标准：

- 平台不只服务个人创作，也能服务开发者、学校、文旅、内容平台。

## 10. 不要现在做的事

这些方向有价值，但不应抢在 MVP 前面：

1. 不要一开始就做复杂交易市场。
2. 不要一开始就上区块链。
3. 不要一开始就做完整音乐版权体系。
4. 不要让 AI 生成作品直接跳过查重。
5. 不要让用户未确认协议就公开发布作品。
6. 不要把用户 API Key 拿去调内部模型。
7. 不要把争议作品永久上链。

## 11. 当前项目下一阶段固定执行方向

结合当前项目状态，后续开发顺序固定为：

1. 保持诗词 API、Qanlo 计费、控制台稳定。
2. 新增用户原创作品库。
3. 新增开放版权协议确认。
4. 新增基础查重。
5. 把当前“意境图 Prompt”升级为作品级生图能力。
6. 再做积分/余额消耗。
7. 最后做交易、音乐、区块链。

这条路线可以把项目从“诗词查询 API”升级成“诗词曲赋创作平台”，同时不会一开始就被交易、区块链、音乐生成这些大模块拖死。

## 12. 外部依据

- [Creative Commons CC0](https://creativecommons.org/publicdomain/zero/1.0/deed.en) 是面向“尽可能放弃版权、贡献到公共领域”的工具，可作为平台开放版权协议的产品参考。
- [Creative Commons Public Domain](https://creativecommons.org/public-domain/) 区分 CC0 与 Public Domain Mark；[PDM](https://creativecommons.org/publicdomain/mark/1.0/deed.en) 更适合标识“已经无已知版权限制”的作品，不能替代用户主动授权协议。
- [WIPO 伯尔尼公约 Article 6bis](https://www.wipo.int/wipolex/en/text/283698) 涉及作者署名和反对有损名誉修改等精神权利，所以平台协议要区分“开放经济性使用”和“不得抄袭、不得冒名、争议可下架”。
